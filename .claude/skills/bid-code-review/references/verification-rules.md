# 検証ルール（全 STEP 共通）

全 STEP に適用する検証ルール集。Step 8 の自己再検証手順は `step-08-self-verification.md` を参照。

---

## パス・ファイル参照の検証（全 STEP で必須）

レビューの指摘・修正例でファイルパスを記載する場合、**必ず実在を確認**してから記載すること。

### 検証ルール

| ルール | 検証方法 | 違反例 |
|--------|---------|--------|
| 指摘・修正例に記載するファイルパスは実在を確認 | `Glob` または `ls` で存在確認 | 存在しないファイルパスを修正例に記載 |
| Grep/Glob のパス引数が正しいか確認 | リポジトリルート vs パッケージルートの起点を意識 | 起点を取り違える |
| import パスとファイルシステムパスを混同しない | Python の `from src.domain.service import ...` はファイルパス `src/domain/service/` に対応するが同一ではない | import パスをそのまま `Read("src.domain.service")` に渡す |
| 修正例中の参照先ファイルの存在と export を確認 | 修正例提示前に確認 | 存在しないモジュールからの import を修正例に含める |

### 検証手順

1. **指摘記述前**: `Glob` でパスの実在確認
2. **Grep/Glob 実行前**: パス引数の起点確認（現在作業ディレクトリを意識）
3. **修正例記述前**: 修正例中の import 文・参照パスが実在するか確認

## 外部依存の最新情報確認（必須）

GitHub Actions バージョン、外部ライブラリバージョン、API 仕様など**頻繁に更新される外部情報**は以下を厳守:

1. **トレーニングデータの知識のみで「存在しない」「非推奨」「最新は vX」と断定しない**
2. **指摘前に必ず最新情報を確認**:

| 対象 | 確認コマンド |
|------|------------|
| GitHub Actions バージョン | `gh api repos/{owner}/{repo}/releases/latest --jq '.tag_name'` |
| GitHub Actions タグ一覧 | `gh api repos/{owner}/{repo}/git/refs/tags --jq '.[].ref' \| sort -V \| tail -5` |
| Python パッケージ | `uv pip show {package}` |

3. **確認できなかった場合**:
   - 「存在しない」と断定せず、**「確認推奨」として Suggestion** で出力する
   - Critical / Major にしてはならない

## 外部ツール設定の仕様検証（必須）

外部ツール（GitHub Actions, release-drafter, labeler 等）の設定ファイルをレビュー・修正提案する際は、**設定キーの仕様を必ず検証**してから指摘・提案すること。

### 禁止される行為

| 禁止行為 | 具体例 | 正しい対応 |
|---------|--------|-----------|
| 設定キーの動作を推測で断定する | `_extends` がファイルパスを参照すると仮定して修正提案 | ドキュメントまたはソースで `_extends` の仕様を確認 |
| 症状のみ修正し根本原因を放置する | パスエラーを直すがそもそも機能がサポートされていない | エラーメッセージの背後にある仕組みを理解してから修正 |

### 検証手順

1. **公式ドキュメントの確認**: ツールの README / 設定リファレンスで該当キーの仕様を確認
2. **同一リポジトリ内の実績確認**: 同じツールが正常動作している他の設定ファイルを参照
3. **仕様が確認できない場合**: 「仕様確認が必要」と明示し、推測に基づく修正提案をしない
