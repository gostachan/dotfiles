# Step 9: Codex 補完レビュー（クロスチェック）

Step 8 までに整理した Claude のレビュー結果ドラフトを Codex CLI（OpenAI Codex CLI）に渡し、見落とし・反対意見・補強観点を取得する補完レビュー手順。

**前提**: このステップは Step 8 完了後・最終出力直前に実行する。Codex に渡すのは Step 8 で整理済みのクリーンドラフトであり、未整理のドラフトを渡さないこと。

---

## 9.1 codex CLI の存在確認

Bash で以下を実行し、終了コード 0 でない場合は Step 9 全体を完全スキップする。

```bash
command -v codex >/dev/null 2>&1
```

スキップ時は最終出力フォーマットの「Codex 補完レビュー」セクションに以下を記載:

```
**実行ステータス**: ⏭ スキップ（理由: codex CLI 未インストール）
```

Step 1〜8 の成果は通常通り出力する。codex CLI 不在はメインフローを止めない。

---

## 9.2 メタ循環回避の対象除外

Codex に渡す変更ファイル一覧・差分本文から `.claude/**` 配下のファイルを除外して Step 9 を実行する。Step 9 全体はスキップしない。

理由: コードレビュースキル（`.claude/skills/bid-code-review/**`）の改修自体を Codex に評価させると、レビューロジックがレビュー対象に含まれる循環が発生する。一方、`.claude/**` を入力から除外することで、混在 PR（例: `.claude/**` + `src/**`）でも `src/**` 等の本体コードには Codex 補完が当たり、メタ循環は依然として回避できる。

### エッジケース: 除外後にレビュー対象が 0 件

`.claude/**` を除外した結果、変更ファイルが 0 件になる場合は Step 9 をスキップする。Codex に渡すレビュー対象が存在しないため、実行する意味がない。

スキップ時は最終出力に以下を記載:

```
**実行ステータス**: ⏭ スキップ（理由: レビュー対象がすべて `.claude/**` 配下のため除外後に空）
```

## 9.3 Codex に渡すプロンプトの構築

プロンプトは Bash のヒアドキュメント経由で組み立てる（一時ファイルにレビュー結果を書かない）。

### プロンプトに含める要素

1. **役割宣言**: 「Claude のレビュー結果を補完する第二レビュアー」
2. **レビュー観点**: Step 2〜8 の観点リスト（クリーンアーキテクチャ整合性 / Python コーディングルール / ロジックバグ / セキュリティ / エラーハンドリング / 影響範囲 / テスト網羅性 / PRD 整合性 / Design Doc 整合性）
3. **Claude のレビュー結果ドラフト**: Step 8 までに整理済みの最終形に近いドラフト全文
4. **変更ファイル一覧**: `git diff --name-only -- ':!**/.claude/**'` の出力（§9.2 のメタ循環回避により `.claude/**` 配下を除外）
5. **差分本文**: `git diff -- ':!**/.claude/**'` の本文（同じく `.claude/**` を除外）。1 万行を超える場合は除外後の先頭 8000 行に切り詰め、末尾に `[truncated]` と記す
6. **出力指示**: `(A) 見落とし` `(B) 反対意見` `(C) 補強観点` の 3 カテゴリで Markdown 出力させる

### プロンプトテンプレート

```text
あなたは Claude のレビュー結果を補完する第二レビュアーです。
Claude が見落とした観点・Claude の指摘への反対意見・Claude の指摘を補強する観点の 3 カテゴリで指摘してください。

## レビュー観点（Claude と同じ観点で補完してください）

- クリーンアーキテクチャ整合性（依存方向 / ファイル配置 / インターフェース整合性 / simulation モジュール同期）
- Python コーディングルール（型ヒント / コメント・docstring / 命名規則）
- ロジックバグ（条件分岐 / 外部接続 / 予算・tCPA 計算）
- セキュリティ（SQL インジェクション / 機密情報 / パストラバーサル）
- エラーハンドリングの網羅性
- 影響範囲（呼び出し元 / モノレポ他パッケージ / simulation モジュール）
- テストパターン網羅性（正常系 / 異常系 / 境界値 / テーブル駆動準拠）
- PRD との整合性（`docs/requirements/*_PRD.md`：機能要件・受け入れ基準）
- Design Doc との整合性（`docs/design-doc/*.md`：技術仕様・データモデル・処理フロー）

## 重要な制約

- Claude が既に指摘済みの内容は **省略**してください（重複は不要）
- 重大度は Critical / Major / Minor / Suggestion の 4 段階で
- 推測や将来可能性のみの指摘は最大 Minor に留めること
- 根拠（ファイルパス・行番号）を必ず明記すること

## Claude のレビュー結果ドラフト

{Claude のレビュー結果ドラフト全文}

## 変更ファイル一覧

{git diff --name-only -- ':!**/.claude/**' の出力}

## 差分本文

{git diff -- ':!**/.claude/**' の本文（必要に応じて除外後の先頭 8000 行に切り詰め）}

## 出力フォーマット

以下の 3 カテゴリで Markdown 出力してください。該当なしのカテゴリは「該当なし」と明記してください。

### (A) Claude が見落とした指摘
- [重大度] file.py:行 — 内容（Claude が見落とした理由・あなたの根拠）

### (B) Claude の指摘への反対意見
- [重大度] file.py:行 — Claude の指摘要約 / あなたの反論

### (C) Claude の指摘を補強する観点
- [重大度] file.py:行 — 内容
```

## 9.4 codex 実行コマンド

実機検証済みのコマンド構成:

```bash
PKG_ROOT="$(git rev-parse --show-toplevel)/packages/bid_allocation_agent"
CODEX_OUT=$(mktemp)
CODEX_LAST=$(mktemp)
trap 'rm -f "$CODEX_OUT" "$CODEX_LAST"' EXIT  # 一時ファイルを確実に削除

# $PROMPT は §9.3 のテンプレートに沿ってヒアドキュメント等で構築済みであること。
# 認証情報・トークンを含む差分をプロンプトに含める場合は、本変数構築前に
# `sed -E 's/(token|password|secret)=[^[:space:]]+/\1=***/gi'` 等でマスクすること（§9.7 参照）。
perl -e 'alarm shift; exec @ARGV' 180 \
  codex exec \
    --skip-git-repo-check \
    --sandbox read-only \
    --color never \
    -C "$PKG_ROOT" \
    -o "$CODEX_LAST" \
    "$PROMPT" </dev/null >"$CODEX_OUT" 2>&1
STATUS=$?
```

### 一時ファイルの役割分担

| 変数 | 内容 | 利用タイミング |
|------|------|---------------|
| `$CODEX_LAST` | `-o` で codex が書き出す最終応答 1 メッセージ | 正常終了時のレビュー本文として利用（§9.6） |
| `$CODEX_OUT` | codex プロセスの stdout/stderr 全体 | エラー終了時の診断ログとして §9.5 の出力に添付（exit code 非ゼロ時のみ） |

### オプションの根拠

| オプション | 役割 |
|-----------|------|
| `--skip-git-repo-check` | `-C` で指定した package root（`packages/bid_allocation_agent/`）は monorepo のサブディレクトリで `.git` を直接持たないため、codex デフォルトの「現在地が git リポジトリ起点か」チェックでの誤検出を回避する |
| `--sandbox read-only` | ファイル書き込みを禁止（レビューは読み取り専用で十分） |
| `--color never` | ターミナルエスケープシーケンスを除去 |
| `-C <package_root>` | 作業ディレクトリを `packages/bid_allocation_agent/` に設定 |
| `-o <file>` | Codex の最終応答 1 メッセージのみを単独ファイルに保存 |
| `</dev/null` | stdin を閉じて追加入力待ちを回避 |
| `perl -e 'alarm shift; exec @ARGV' 180` | macOS には GNU `timeout` 不在のため Perl で代替（タイムアウト 180 秒） |

## 9.5 失敗時の振る舞い

判定は以下の順序で行う。`142` は `!= 0` にも該当するため、**必ず `142` を先に判定**する。

| 判定順 | ケース | 判定式（例） | 動作 |
|-------|-------|------------|------|
| 1 | タイムアウト | `[ "$STATUS" -eq 142 ]` | 最終出力に `Codex レビューが 180 秒でタイムアウトしたためスキップしました` を記録 |
| 2 | その他のエラー | `[ "$STATUS" -ne 0 ] && [ "$STATUS" -ne 142 ]` | 最終出力に `Codex レビューはエラー終了したためスキップしました（exit=$STATUS）` を記録し、`$CODEX_OUT` の末尾抜粋（最終 30 行程度）を併記する |
| 3 | 空応答 | `[ "$STATUS" -eq 0 ] && [ ! -s "$CODEX_LAST" ]` | 最終出力に `Codex から有効な応答が得られませんでした` を記録 |

いずれの場合も Step 1〜8 の成果は通常通り出力する。

`$CODEX_OUT` の末尾抜粋は **判定処理内（trap EXIT 発火前）** に `tail -n 30 "$CODEX_OUT"` で取得し、最終出力用の変数に格納しておくこと。

## 9.6 Codex 出力の統合

`$CODEX_LAST` の Markdown を最終出力フォーマットの「Codex 補完レビュー」セクションに反映する。

### Claude による採否判定（必須）

Codex の指摘を**そのまま**最終出力に貼り付けず、Claude が各指摘について採否を判定し「Codex 指摘の最終判定」テーブルに明示する。

| 区分 | 意味 |
|------|------|
| ✅ 妥当 | Codex の指摘は正しく、ユーザーが対応すべき |
| ❌ 不採用 | Codex の指摘は誤り（推測・誤読・到達不能パス等） |
| ⚠️ 要追加調査 | Codex の指摘は妥当性が即断できず、ユーザーに判断を仰ぐ |

---

## 9.7 セキュリティ・データ保護

- Codex に渡すプロンプトには**ソースコード差分とレビュー結果のみ**を含める
- 認証情報・トークン・シークレットを含む変更がある場合、該当行を `***` でマスクしてからプロンプトに含めること
- `--sandbox read-only` により Codex がファイルを書き込むことはないが、念のためプロンプトには「変更を提案するのみで、ファイル編集を行わないこと」と明記する
