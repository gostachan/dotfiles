# Step 10: GitHub PR インラインコメント投稿

Step 1〜9 で収集した指摘を GitHub PR の Files Changed タブにインラインコメントとして投稿し、レビュアーが diff 上で直接確認できるようにする手順。**自動投稿はせず、必ずユーザー承認を経てから投稿する**。

**前提**: このステップは PR モード（`$ARGUMENTS` が PR 番号）かつ Step 8 完了後・最終出力後に実行する。ブランチモード・ファイルモード・現在ブランチモードでは完全スキップする。

---

## 10.1 実行条件

`$ARGUMENTS` が PR 番号の場合のみ起動する。それ以外のモードでは完全スキップし、最終出力にも「Step 10」セクションを出さない。

```bash
# 例: PR 番号判定
if [[ "$ARGUMENTS" =~ ^[0-9]+$ ]]; then
  PR_NUMBER="$ARGUMENTS"
else
  # Step 10 スキップ
  exit 0
fi
```

---

## 10.2 指摘抽出

Step 8 の最終出力（自己再検証済みのレビュー結果）から、インラインコメントとして投稿可能な指摘を抽出する。

抽出対象に含めるフィールド:

| フィールド | 必須 | 備考 |
|------------|------|------|
| ファイルパス | ✅ | PR の変更ファイルに含まれていること |
| 行番号 | ✅ | 単一行を指す。範囲指定（start_line + line）は §10.5 を参照 |
| 重大度 | ✅ | Critical / Major / Minor / Suggestion のいずれか |
| 指摘本文 | ✅ | 根拠と推奨対応を含む |

### 抽出対象から除外する指摘

- 行番号が存在しない指摘（ファイル全体・モジュール全体・アーキテクチャ観点など) → 通常のテキスト出力のみで対応
- PR の変更ファイル外を参照する指摘 → 「該当行が PR に含まれない」ためインラインコメント不可（テキスト出力のみで対応）

---

## 10.3 投稿プレビューとユーザー承認【必須】

`gh api` を呼ぶ前に、投稿予定のコメント一覧をユーザーに提示し `AskUserQuestion` で承認を取得する。**自動投稿は禁止**。

### プレビュー表示フォーマット

```
## 投稿プレビュー: GitHub PR #{PR_NUMBER} インラインコメント

投稿予定: {N} 件
- 🔴 Critical: {a} 件
- 🟠 Major: {b} 件
- 🟡 Minor: {c} 件
- 💡 Suggestion: {d} 件

### コメント一覧
1. [{重大度}] {ファイルパス}:{行番号}
   > {指摘本文の冒頭}

2. [{重大度}] {ファイルパス}:{行番号}
   > {指摘本文の冒頭}

...
```

### AskUserQuestion で確認する選択肢

| 選択肢 | 動作 |
|--------|------|
| 投稿する | `gh api` で一括投稿（§10.4） |
| 編集する | 除外したい指摘や本文修正をユーザーから受け取り、再度プレビューしてから承認を取り直す |
| 取りやめる | 投稿せず終了。最終出力に「ユーザー承認が得られなかったため投稿スキップ（テキスト出力のみ）」を記載 |

---

## 10.4 投稿コマンド

ユーザーが「投稿する」と明示的に承認した場合のみ実行する。

```bash
OWNER=$(gh repo view --json owner --jq '.owner.login')
REPO=$(gh repo view --json name --jq '.name')
PR_NUMBER={$ARGUMENTS から抽出した PR 番号}

gh api "repos/${OWNER}/${REPO}/pulls/${PR_NUMBER}/reviews" \
  --method POST \
  --input - <<'PAYLOAD'
{
  "body": "Claude Code による自動コードレビュー（/bid-code-review スキル）",
  "event": "COMMENT",
  "comments": [
    {
      "path": "{ファイルパス}",
      "line": {行番号},
      "side": "RIGHT",
      "body": "{重大度アイコン} **{重大度}**: {指摘内容}"
    }
  ]
}
PAYLOAD
```

### コメント本文フォーマット

```
🔴 **Critical**: {指摘内容}
🟠 **Major**: {指摘内容}
🟡 **Minor**: {指摘内容}
💡 **Suggestion**: {指摘内容}
```

重大度アイコンと「**{重大度}**:」プレフィックスを必ず付与する。レビュアーが Files Changed タブで一目で重大度を判別できるようにするため。

---

## 10.5 複数行コメント（範囲指定）

複数行にまたがる指摘は `start_line` と `line` を併用して範囲を指定する。`side`・`start_side` は両方とも `RIGHT` を指定する。

```json
{
  "path": "{ファイルパス}",
  "start_line": {開始行},
  "start_side": "RIGHT",
  "line": {終了行},
  "side": "RIGHT",
  "body": "{重大度アイコン} **{重大度}**: {指摘内容}"
}
```

---

## 10.6 失敗時の振る舞い

| ケース | 判定 | 動作 |
|--------|------|------|
| 指摘が 0 件 | 抽出結果が空 | プレビュー・承認も不要。最終出力に「指摘 0 件のためインラインコメントなし」を記載 |
| ユーザーが「投稿する」以外を選択 | `AskUserQuestion` の応答が「編集する」「取りやめる」 | 編集要望は反映して再プレビュー、取りやめは終了して最終出力に「ユーザー承認が得られなかったため投稿スキップ」を記載 |
| `gh api` 失敗（認証エラー・権限なし等） | `gh api` の exit が非ゼロ | エラー内容をユーザーに報告して続行。テキスト出力のレビュー結果は既に出力済みのため処理は止めない |
| 一部の指摘で行番号が PR の変更範囲外 | GitHub API が 422 を返す | 該当指摘のみ除外して再投稿。除外した指摘はテキスト出力のみで残す |

いずれの場合も Step 1〜9 のテキスト出力は通常通り提供する。Step 10 はオプショナル扱い。

---

## 10.7 セルフチェックポイント

- [ ] PR モード（`$ARGUMENTS` が PR 番号）であることを確認した
- [ ] 投稿前にプレビューと件数をユーザーに提示し、`AskUserQuestion` で承認を取得した
- [ ] ユーザーが「投稿する」と明示的に承認した、または投稿スキップを記録した
- [ ] `gh api` 実行時は exit=0 で完了した、または失敗理由をユーザーに報告した
- [ ] 投稿したコメント数（または投稿スキップ）をユーザーに報告した

---

## 10.8 禁止事項

- **ユーザー承認なしの自動投稿**: 誤指摘や余計な指摘が投稿されると PR の見通しが悪化する。必ず `AskUserQuestion` で承認を取得する
- **PR モード以外で `gh api` を呼ぶ**: 存在しない PR 番号への投稿試行でエラーになる
- **`gh api` 失敗でレビュー全体を中断**: テキストレビュー結果が失われる。失敗はエラー報告のみとし、テキスト出力は継続する
- **行番号なしの指摘をインラインコメントとして投稿**: GitHub API がエラーを返すか、不正な位置にコメントされる。行番号がない指摘はテキスト出力のみで対応する
