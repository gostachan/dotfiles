# Step 8: 自己再検証（内部作業）

Step 1〜7 の結果をまとめた**後**、最終出力する**前**に実施。**内部作業でありユーザーに出力しない。**

---

## 8.1 各指摘の実行パス検証

- [ ] 指摘した問題のコードパスに**到達可能か**（到達不能コードへの指摘は無効）
- [ ] 指摘した条件が**実際のデータ・呼び出し元で発生し得るか**
- [ ] 「将来〜かもしれない」だけで根拠が推測に留まっていないか

## 8.2 重大度の妥当性チェック

| 重大度 | 基準 | 許容される根拠 |
|--------|------|---------------|
| Critical | セキュリティ脆弱性、データ損失、致命的バグ | 現在のコードで**確実に発生する**問題のみ |
| Major | 動作上の不具合、エラーハンドリング漏れ | 現在のコードで**到達可能なパス**に基づく問題 |
| Minor | コード品質、可読性、保守性 | 改善が望ましいが動作に影響しない問題 |
| Suggestion | 提案、代替案 | 現状でも問題ないが、より良くできる点 |

**再分類ルール**:
- 「将来〜になるかもしれない」→ 最大でも Minor
- 「到達可能だがユーザーに致命的影響はない」→ Major に留める
- 「コメント追加で十分」→ Suggestion

## 8.3 パス・参照の正確性チェック

- [ ] 指摘で言及したファイルパスが全て実在するか（`Glob` で確認済みか）
- [ ] 修正例の import 文が実在するモジュール・クラスを参照しているか
- [ ] Grep/Glob コマンドのパス引数が作業ディレクトリに対して正しいか

## 8.4 重複・矛盾の排除

- [ ] 同じ問題を異なる観点から重複指摘していないか
- [ ] ある指摘と別の指摘が矛盾していないか
